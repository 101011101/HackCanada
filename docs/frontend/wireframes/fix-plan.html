I-01
humidity silently dropped by POST /suggestions
High Backend + Frontend
Impact
The user enters a humidity value in AddPlotSheet, it is passed through location.state to Suggestions.tsx and included in the getSuggestions() call body. However, backend SuggestionRequest has no humidity field — it is silently dropped at the HTTP boundary. The route handler hardcodes humidity=60.0 for the temporary FarmNode used by the scorer. The scorer therefore always evaluates crop suitability against 60% humidity regardless of what the user reported. Crops that require low or very high humidity environments will be ranked incorrectly for every user.

I-02
FarmContext single-farm: join() silently overwrites existing farm
High Architecture
Impact
A joined user who opens AddPlotSheet from the Dashboard, completes the form, and confirms crop selection in Suggestions.tsx triggers join(farmId, lat, lng) which overwrites mycelium:farm_id, mycelium:farm_lat, mycelium:farm_lng in localStorage with their new plot's values. Their original farm is lost without warning. The Dashboard then fetches data for the new farm only. The user cannot return to their previous farm. This breaks the "add a second plot" use case entirely. Additionally, the context stores no farm list, so there is no way to switch between farms.
Architectural decision
Option A (recommended — minimal): Keep single-farm storage but show a confirmation modal before overwriting. Make the intent explicit: "This will replace your current farm. Continue?" This preserves current data shape, requires no migration, and matches hackathon scope.

Option B (future): Store a farm list in localStorage and support switching. Larger refactor — out of scope for now.

I-03
lat/lng fallback is 0,0 (Atlantic Ocean) for POST /nodes
High Data Integrity
Impact
AddPlotSheet provides a "Use my location" GPS button but it is optional. When the user skips GPS, form.lat and form.lng remain null. Suggestions.tsx line 162 uses formData.lat ?? 0 and formData.lng ?? 0 as fallback to createNode(). The backend NewFarmRequest requires lat: float and lng: float (non-optional). So lat=0, lng=0 (the Gulf of Guinea / Atlantic Ocean) is stored for the farm node. Any geo-aware features (routing to hubs, distance calculations, regional scoring) will silently fail or produce nonsense results for all plots created without GPS.

I-04
useAsync fires on mount — architectural misuse for POST requests
High Hook Misuse
Impact
useAsync always runs fn() on mount via useEffect(() => { run(); }, [run]). Suggestions.tsx currently uses useAsync to call getSuggestions() — a POST request — immediately on render. This works in the happy path (formData is available from router state), but causes two concrete problems:

1. Race on hot-reload / React StrictMode: the effect fires twice in development, sending two simultaneous POST requests. While results are deduplicated by the counter ref, it is wasteful and confusing in DevTools.

2. Architectural debt: if formData ever changes (user hits back, modifies form, returns), the suggestions re-fetch automatically via the [JSON.stringify(formData)] dep. This is accidentally correct but fragile. The pattern is not reusable for actual imperative POSTs (e.g. createNode already correctly uses plain async/await). Future contributors will cargo-cult this pattern.

The fix does not require removing the useAsync call — but it should be replaced with useState + useEffect triggered explicitly, mirroring the pattern used in Setup.tsx.

I-07
ReadingEntryResponse field mismatch between backend and frontend
Medium Contract Mismatch
Impact
Backend ReadingEntryResponse (models.py lines 46–52) emits: farm_id, timestamp, pH, moisture, temperature, humidity. Frontend type (types.ts lines 96–104) expects: id, node_id, pH, moisture, temperature, humidity, recorded_at.

DataTrendsChart likely uses recorded_at as the x-axis timestamp — if so, the chart renders with no time axis data, showing all points at x=undefined (flat or missing line). postReadings() response is unused beyond logging, so that side is less critical.

I-09
/suggestions route is publicly accessible without auth
Medium Routing
Impact
App.tsx places /suggestions outside of ProtectedRoute (line 32), so any unauthenticated user can navigate to it. The null-guard in Suggestions.tsx handles formData === null gracefully — but only shows a static fallback message with a "Back to dashboard" button, which then bounces them to / (since they are not joined). Additionally, SetupRoute correctly redirects joined users away from /setup to /dashboard. AddPlotSheet is already invoked from Dashboard so it does not depend on the setup route — no issue there. The medium risk is a developer who deep-links to /suggestions in production sees a confusing fallback rather than a meaningful redirect.
