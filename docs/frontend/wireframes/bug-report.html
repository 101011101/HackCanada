01
Duplicate React key warning in CropPickerSheet
High Frontend · React UI Corruption
Symptom

React console warning: "Encountered two children with the same key 1. Keys should be unique so that components maintain their identity across updates." The CropPickerSheet renders incorrectly — React cannot distinguish between the two crop rows, causing state confusion on re-renders (wrong checkmark positions, potential flicker).

Root Cause

File: app/frontend/src/user/components/sheets/CropPickerSheet.tsx, line 64.

The bundles.map() uses key={bundle.crop_id}. The crop_id values in BundleResponse come from CROP_TASKS keys in the backend, which are zero-based integers (0, 1, 2 …). When two crops are assigned, the bundles array returned by GET /nodes/{farm_id} contains two BundleResponse objects. If the farm was assigned crop index 1 (Lettuce) as both its first and second slot (or any two crops that share a crop_id), the key collision fires.

The real issue is more subtle: the key is crop_id, not farm_id + crop_id. For a single-farm user the farm_id is always the same, so if the engine ever assigns the same crop_id twice (a known bug in the multi-slot greedy path where modified[already] = 0.0 only blocks already-assigned indices within the same call — not across calls), the picker will have two rows with the same key.

More practically: the bundles array is fed directly from nodeState.data which comes from GET /nodes/{id}. That endpoint loops over assignments[farm_id] which is a list of crop indices. If the greedy algorithm placed the same crop index twice in the list (which the deduplication logic in _greedy_assign is supposed to prevent, but does not prevent across two separate preferred_crop_ids calls), two bundles with identical crop_id appear in the array and both render with key=1.

Minimal Fix

Change the key prop on line 64 from bundle.crop_id to a composite of farm_id and crop_id:

Before: key={bundle.crop_id}

After:   key={`${bundle.farm_id}-${bundle.crop_id}`}

This makes the key unique even if the same crop appears twice on the same farm. It also future-proofs multi-farm UIs. No other component changes are needed — the onSelect callback already uses bundle.crop_id, which remains correct.

A deeper fix would also deduplicate the assignments list in the backend's _greedy_assign function (nodes.py line 64) before saving, ensuring a farm cannot be assigned the same crop index twice — but that is a separate backend concern and not strictly required to silence the React warning.

02
GET /nodes/{id}/tasks returns 100+ tasks when two crops are selected
Critical Backend · Python Data Correctness
Symptom

After the user selects two crops on the Suggestions page and is redirected to the Dashboard, GET /nodes/23/tasks returns a massive array — well over 100 items. The Dashboard task list renders every task from every crop in the system, not just the two chosen ones.

Root Cause

File: app/backend/api/routes/nodes.py, lines 410–411.

The task endpoint reads the farm's assignment from assignments.json via storage.load_assignments(). That function (storage.py line 68–74) applies a migration shim that converts the old scalar format to a list: v if isinstance(v, list) else [v]. This is fine for existing farms.

The problem is in add_node (nodes.py line 147–148). The greedy assignment returns assigned, which is a list of crop array indices (0-based positions into the crops list). These indices are the same as crop IDs only because crops[0].id == 0, crops[1].id == 1, etc. That accidental alignment holds for the seed data, but it means assignments[str(new_id)] = assigned stores indices, not IDs.

When two crops are selected via preferred_crop_ids, the greedy algorithm fills two slots. With 10 crops in the system and a large plot, num_slots can be much larger than 2. The code on line 55–68 runs for slot in range(num_slots) and appends a crop index for every slot. For a 200 sqft plot with min_slot_sqft=20 that is 10 slots, each assigning a crop index. The assignments list ends up as e.g. [1, 3, 0, 5, 2, 1, 3, 0, 5, 2] — 10 entries, not 2.

Back in get_farm_tasks (nodes.py line 410), crop_ids = assignments.get(str(farm_id)) returns this full 10-element list. The loop on line 422 iterates over every entry and fetches all tasks for that crop. Because the list contains duplicates (same crop index 3 times), each crop's task list is appended multiple times. Lettuce has 6 tasks × appearing 3 times = 18 Lettuce tasks; multiply across all 10 slot assignments and you get 60–100+ task items.

Secondary Collision: TaskList key={task.id}

The task objects emitted by the backend reuse the IDs defined in CROP_TASKS. Each crop's task list is numbered id: 1 through id: 6 (or 7) independently. When two crops are assigned, both crops have a task with id: 1, id: 2, etc. TaskList.tsx line 40 uses key={task.id}, so even if the duplicate-crop issue is fixed in the backend, a two-crop farm will still generate the same "Encountered two children with the same key" warning in TaskList — a different location than the CropPickerSheet warning but the same React error message. The task IDs are only unique within a crop, not across crops.

Minimal Fix

Fix A — Backend (primary fix): In get_farm_tasks (nodes.py line 421), deduplicate crop_ids before iterating:

Before: for crop_id in crop_ids:

After:   for crop_id in dict.fromkeys(crop_ids):  # preserves order, removes duplicates

This ensures each crop's tasks are added only once regardless of how many slots in the assignment list point to the same crop. The same one-liner should also be applied in get_node (line 273) and get_guide (line 388) for consistency.

Fix B — Frontend (secondary fix for TaskList key collision): In TaskList.tsx line 40, change the key to a composite of crop and task IDs:

Before: key={task.id}

After:   key={`${task.crop_id}-${task.id}`}

This is a defensive fix. Even with Fix A in place, the task IDs are semantically scoped to a crop (Lettuce task 1, Tomato task 1 are different tasks), so the composite key is more correct regardless.

Root-cause fix (optional hardening): Store only unique crop IDs in assignments.json at save time in add_node line 148: assignments[str(new_id)] = list(dict.fromkeys(assigned)). This prevents the duplication from ever reaching the database.

Trigger condition
Both bugs fire the moment a user selects 2+ crops on the Suggestions page and POST /nodes is called with preferred_crop_ids of length ≥ 2.
